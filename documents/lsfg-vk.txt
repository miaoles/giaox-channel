================================================================================
GIAOX — LSFG-VK INTEGRATION ROADMAP
================================================================================
OVERVIEW
    Seven phases move every lsfg-vk concern out of the home configuration
    and into the component that owns it: the package owns the artifact,
    a home service owns policy and configuration, the machine layer owns
    hardware-specific values, and nonguix' Steam container owns sandbox
    reachability.
    The guiding rule is DEFAULT-UPSTREAM, OPT-IN GUIX: with no
    configuration beyond installing the package, lsfg-vk must behave
    exactly as its authors intend — implicit Vulkan layer, profile-based
    activation via active_in, conf.toml created by the layer and owned by
    lsfg-vk-ui.  Every deviation that makes it more declarative, more
    inert, or more reproducible is something the maintainer opts into.
    Work proceeds per documents/conventions.txt in the giaox repository:
    one step at a time, full-file delivery, a single verification bucket
    per step, maintainer verification before the next step begins.
    Phase order stands as written.  Phases 4, 5 and 6 are independent of
    one another and may be reordered or dropped at the maintainer's
    discretion.
OWNERSHIP (reference)
    PACKAGE   giaox-channel: giaox/packages/lsfg-vk.scm
        Builds the layer, the CLI and the UI.  Guarantees the layer
        manifest is discoverable and that its library_path resolves.
        Publishes the constants other layers need (layer name, the
        environment-variable regexp set) so nothing downstream
        rediscovers them by globbing the output.  Also publishes the
        container transformer and the Steam constructor, because making a
        Vulkan layer reachable inside an FHS container is a property of
        the layer, not of Steam — the same reasoning by which nonguix puts
        steam-nvidia in its nvidia module.
        Contains NO site policy, NO paths belonging to the configuration
        repository, NO machine-specific values, and NO knowledge of any
        particular GPU driver: the driver and its passthrough set are
        required positional parameters with no defaults.
    SERVICE   giaox-channel: giaox/services/lsfg-vk.scm
        A HOME service: everything it does is per-user — it writes
        $XDG_CONFIG_HOME/lsfg-vk/conf.toml, which upstream documents as
        the config path and lsfg-vk-ui owns, sets session variables, and
        populates a profile.  Placed beside the channel's existing
        file-organizer home service rather than in a new home/services
        subtree, per maintainer decision.  See D8 for how the system
        variant stays possible without a refactor.
    MACHINES  giaox: machines/*/main.scm
        The GPU identity, the Steam variant, the graphics driver, and the
        driver's sandbox passthrough set.
    HOME      giaox: home/main.scm
        One service instantiation carrying the maintainer's profiles and
        the path to Lossless.dll.  Nothing else.
KEY INVARIANTS (reference)
    I1  The layer manifest's library_path is an absolute store path, and
        that path resolves inside a nonguix container EXACTLY WHEN the
        package is a member of that container.  The launcher runs
        `guix shell --container -m <manifest>', which mounts the closure of
        the container manifest — not the whole store.  Membership and
        path validity are therefore one fact, not two.  Nothing may ever
        rewrite this path; rewriting it is what the superseded staging
        machinery did, and the staging existed only because the package
        was not a container member.
    I2  The FHS union is mounted at /usr inside the container, so the
        Vulkan loader's default search path reaches the union's
        share/vulkan/implicit_layer.d without any environment variable.
        Container membership replaces file staging entirely.  Confirmed by
        observation: nvidia_layers, MangoHud and VkLayer_MESA_device_select
        are already present there and already point into store paths
        reached through the union's references.
    I3  Environment variables are the ONLY thing that must cross the
        sandbox boundary explicitly, via the container's preserved-env.
        The complete set lsfg-vk 2.0 reads is DISABLE_LSFGVK plus ten
        LSFGVK_* variables; "^DISABLE_LSFGVK$" and "^LSFGVK_" therefore
        cover it exactly.  Any variable this project introduces must fall
        under the LSFGVK_ prefix so the regexp set never grows.
    I4  conf.toml is mutable state written by lsfg-vk-ui and by the layer
        itself.  The default configuration must never make it read-only.
    I5  lsfg-vk is x86_64-only.  This is DECLARED but NOT ENFORCED:
        (supported-systems '("x86_64-linux")) governs the CLI, and a
        consumer that passes #:system explicitly — nonguix' fhs-union does —
        never consults it.  Any 32-bit consumer must therefore be excluded
        by construction, not by the field.  See I10.
    I6  The channel is consumed by `guix pull', so `guix build lsfg-vk'
        exercises the PULLED revision, not the working tree.  Every
        verification of an unpushed change must go through
        `guix build -L <channel>' / `guix shell -L <channel>' /
        `guix repl -L <channel>' / `guix system build -L <channel>'.  A
        bucket that omits -L proves nothing about the file under review.
        COROLLARY: when a -L load fails, `guix shell' warns and silently
        falls back to the PULLED package.  Any result obtained after such a
        warning must be attributed with care.
    I7  The LAYER must resolve the Vulkan loader by bare soname; only the
        EXECUTABLES may be pointed at a specific one.  lsfg-vk-common
        dlopens "libvulkan.so.1" and is a STATIC library linked into all
        three targets, so any patch to that source reaches the layer.  For
        the layer the bare soname is correct and required: it matches the
        loader already present in the application's process, which inside
        the Steam container is the FHS union's, and two loader instances in
        one process is not a working setup.  For the executables no loader
        is present and the soname resolves against nothing, because
        ld-wrapper adds RUNPATH only for linked libraries and a dlopened
        library is never linked.  The fix is therefore scoped to
        executables through the linker, never through the source.
        VERIFIED by objdump: the layer's RUNPATH is glibc and gcc only;
        both executables carry vulkan-loader.
    I8  Verification buckets must not depend on anything unverified about
        the environment.  Three rounds were lost to this: patchelf is NOT
        installed (objdump, nm and strings are); define*-public is NOT bound
        in this Guile/Guix context; and `vkcube' prints its own
        "Selected GPU N:" line, which is not lsfg-vk output and was once
        misread as such.  A check that silently yields empty output, or a
        construct assumed to exist, is worse than no check at all.
    I9  A module in this channel may import ONLY what .guix-channel
        declares as a dependency.  `guix pull' compiles the channel with
        just those modules on the load path, and one uncompilable module
        fails the entire channel derivation — there is no containment.
        Neither -L nor `guix repl' reveals the omission, because pulled
        channels are already on the CLI's load path.  Therefore: every new
        cross-channel import requires a .guix-channel change, and the
        verification of any such change MUST include an actual pull.
        VERIFIED: with nonguix declared, `guix pull' builds nonguix.drv
        ahead of giaox.drv and the channel compiles.
    I10 A nonguix container's union32 is a DELAYED field whose default is
        computed from `packages'.  Appending to `packages' therefore adds
        the package to the 32-bit union as well, which for lsfg-vk means an
        i686 build and an i686 qtdeclarative — and that does not build.
        Any transformer that adds a 64-bit-only package to a container MUST
        pin union32 to the value it had on the incoming container.  Heroic
        overrides union32 explicitly and Steam does not, which is why the
        default's derivation is easy to miss.
DECISIONS (locked)
    D1  Activation policies: upstream implicit (default) plus opt-in gated
        and explicit variants, selected by the service, never baked into
        the default package.
    D2  conf.toml ownership defaults to upstream's workflow — the service
        writes nothing; seed and declarative modes are opt-in escalations.
    D3  The home configuration gains machine dispatch (Phase 5).
    D4  The GUI is split into its own output (Phase 6), after behaviour is
        proven.
    D5  nonguix' %nvidia-environment-variable-regexps is private and is
        read with @@ rather than reconstructed.  Rationale: a reconstructed
        copy degrades SILENTLY when nonguix adds a variable — a game quietly
        loses NGX or G-Sync passthrough with no diff to point at — whereas
        @@ fails LOUDLY at evaluation if the binding moves, and loud
        channel-churn failures are a managed quantity in this project
        (channels/channels.scm documents three of them).  @@ on private
        bindings has precedent in Guix proper (gnu/packages/gnuzilla.scm)
        and in nonguix' own nvidia module, so the idiomatic objection is
        weak.  The expression lives in the LAPTOP'S MACHINE FILE, not in
        the package module, because NVIDIA is a machine concern.  The
        permanent fix is a one-line #:export patch sent upstream.
    D6  Module layout follows Guix: packages and services in separate
        modules.  The container transformer and Steam constructor belong
        to the package module (nonguix precedent).  The service is a home
        service at giaox/services/lsfg-vk.scm, beside the channel's
        existing home service, rather than in a new home/services subtree.
    D7  nonguix is declared as a .guix-channel dependency (I9), with an
        introduction so that it is authenticated in the dependency
        position as well as in channels/channels.scm.  Rejected
        alternatives: moving the container code into the configuration
        repository, where `load'ed files escape channel compilation —
        rejected because it splits the design across two repositories and
        demotes reusable library code to configuration glue; and deferring
        the import to runtime via resolve-interface — rejected because it
        hides a real dependency behind a trick.  The accepted cost is that
        this channel's valid window is now the intersection of its guix and
        nonguix windows.
    D8  Home and system service types cannot be unified: the two service
        graphs are disjoint (home-profile-service-type vs
        profile-service-type, home-files-service-type vs
        etc-service-type), and upstream always writes two types when both
        make sense.  Therefore the configuration records, the TOML
        serializer — a plain procedure from configuration to file-like —
        and the policy/package selection are written as shared module-level
        definitions, and only the HOME service type is built.  A future
        system type writing /etc/lsfg-vk/conf.toml is then an addition,
        never a refactor.
ACCEPTED LIMITATIONS
    L1  A 32-bit Vulkan client inside the Steam container reads the
        64-bit layer manifest from the shared /usr/share, fails to load
        the library, and skips the layer, logging a loader error.  This is
        upstream's situation on every distribution, because upstream does
        not architecture-suffix its manifest the way nvidia_layers and
        MangoHud do.  Adding lsfg-vk to union32 is not a remedy (I5, I10).
    L2  On the laptop, machine-system-transform is replace-mesa, which
        grafts mesa to nvda throughout the system closure — including
        lsfg-vk, which pulls mesa through qtbase.  The graft rewrites store
        references inside files, so the layer manifest's library_path
        follows the grafted output.  The system and home profiles therefore
        hold different lsfg-vk store paths on that machine; this is benign
        because only the container's manifest is visible inside the
        container and only the home profile's on the host.
EXTERNAL BLOCKERS
    B1  At current guix revisions `nix' fails its check phase on
        eval-okay-fromTOML-timestamps.  Addressed by a tests-disabled
        variant, (giaox packages nix), selected through nix-configuration.
        STATUS: STILL REQUIRED.  Re-probed: 0 of 1 substitutes on
        substitutes.nonguix.org, bordeaux and ci; stock nix would still be
        built; the running system is on nix-without-tests-2.25.5.  To be
        removed under SECTION 9 cleanup discipline once upstream
        regenerates its expectation.
    B2  i686 `qtdeclarative' does not build at current guix revisions — its
        QmltcTests fail to compile during the build phase.  This is only
        ever reached by pulling a Qt-dependent package into a 32-bit FHS
        union, which I10 forbids; it is recorded so that the symptom is
        recognised if it reappears.

        * Upstream's manifest already carries
          disable_environment = { DISABLE_LSFGVK: "1" }.  2.0 has no
          enable variable; inertness is a property of profile matching.
        * Variables, enumerated from the built binaries: DISABLE_LSFGVK,
          LSFGVK_CONFIG, LSFGVK_DLL_PATH, LSFGVK_ENV, LSFGVK_FLOW_SCALE,

          LSFGVK_PERFORMANCE_MODE, LSFGVK_PROFILE.  No LSFG_* strays.

          DEVELOP off.  LSFGVK_LAYER_LIBRARY_PATH is a CACHE variable and
          is therefore settable; CMAKE_SKIP_RPATH is forced ON by a plain
          set() and is therefore NOT settable from the command line.
        * vkNegotiateLoaderLayerInterfaceVersion is annotated
          visibility("default") in entrypoint.cpp, which overrides the
          target's hidden visibility preset.  No patch is required to
          export it; verified by nm with the patch removed.
        * lsfg-vk-common is a STATIC library (installed as an ARCHIVE) and
          is linked into the layer, the CLI and the UI.  Its
          src/vulkan/vulkan.cpp dlopens "libvulkan.so.1" with a
          "libvulkan.so" fallback.  See I7.
        * Licence: SPDX-License-Identifier GPL-3.0-or-later throughout, so
          gpl3+ is correct.
        * No Khronos headers are vendored; sources include
          <vulkan/vulkan_core.h> and <vulkan/vk_layer.h>, so vulkan-headers
          is a required build-time input that never becomes a runtime
          reference.

          repeated [[profile]] with name, active_in (string OR array),
          gpu, multiplier, flow_scale, performance_mode, pacing.  Upstream
          writes paths and enumerations as single-quoted literals.
        * The binaries also reference /etc/lsfg-vk/conf.toml, so a
          system-wide configuration path exists.  See D8.
    Container surface
        * Record <nonguix-container>, fields: name binary-name version
          wrap-package run wrapper-name manifest-name internal-name
          sandbox-home ld.so.conf ld.so.cache union64 union32
          preserved-env exposed shared modules packages link-files
          home-page synopsis description license.
        * Accessors are named ngc-* and are ALL PUBLIC, including
          ngc-packages, ngc-preserved-env and ngc-union32.  Step 2.2 needs
          no private access; only D5's NVIDIA regexps do.
        * union32 (and union64) are DELAYED fields defaulting to a derivation
          of `packages'; this-nonguix-container exists in the module's
          private bindings, which is the tell.  See I10.
        * (steam-container-for mesa) yields preserved-env '() — the base and
          debug regexps visible in the launcher are appended by the builder,
          not held in the field — and 52 packages.
        * The launcher honours GUIX_SANDBOX_HOME, and DEBUG=1 makes it
          drop into an interactive shell inside the container.
        * GUIX_SANDBOX_EXTRA_SHARES exists as a colon-separated escape
          hatch; the design must not require it.
        * .guix-channel dependency metadata accepts an introduction in the
          form (introduction (channel-introduction (version 0) (commit ...)
          (signer ...))); verified by parsing candidates, then by a pull.
    Environment
        * Lossless.dll is installed at
          $GUIX_SANDBOX_HOME/.local/share/Steam/steamapps/common/Lossless
          Scaling/.  Inside the container that is the default search
          location, so in-sandbox operation needs no dll configuration; on
          the host it is unreachable by default, which is the sole reason
          the service needs a dll-path field.
        * Devices on the laptop, as the Vulkan loader reports them:
          "Intel(R) Iris(R) Xe Graphics (ADL GT2)" (integrated) and
          "NVIDIA RTX A2000 Laptop GPU" (discrete).
    Closed items
        * O1 CLOSED, accepted as correct output: etc/ld.so.cache is a
          glibc-ld.so.cache produced by gnu-build-system's standard
          make-dynamic-linker-cache phase, indexing the inputs' libraries
          (375 entries).  It is not cruft and must not be deleted.  It is
          also inert for our purposes: it contains mesa's libvulkan_*
          drivers but not libvulkan.so.1, and no system-wide
          /etc/ld.so.cache exists for glibc to consult.  I7's linker flag
          is the mechanism that actually matters.
        * O2 CLOSED by mutation: a stale hand-installed manifest at
          ~/.local/share/vulkan/implicit_layer.d/ pinned
          ~/.local/lib/liblsfg-vk-layer.so (a symlink to a superseded
          output) and gated on LSFG_VK, a variable from an even earlier
          experiment.  The loader searches $XDG_DATA_HOME/vulkan
          unconditionally, so the file polluted even --pure shells and was
          in fact the ONLY manifest those shells ever saw.  Both it and the
          symlink were removed.
        * OB1 ACCEPTED as upstream behaviour: when no profile matches, the
          layer declines loader negotiation.  The loader then looks for
          vkGetInstanceProcAddr, does not find it, logs two ERROR lines and
          SKIPS the layer.  Clients are unaffected and exit cleanly, so
          this is noise rather than breakage — but every Vulkan client on a
          machine with lsfg-vk installed and idle logs those two errors.
          Nothing in the package can change it.  It is a concrete argument
          for Phase 4's policies for anyone who wants clean loader logs.
        * OB2 CLOSED AND CONFIRMED as a GPU mismatch, by experiment:
            vkcube --gpu_number 0 (Intel iGPU), no selector — exit 0, clean.
            vkcube --gpu_number 1 (NVIDIA dGPU), no selector — hang,
              preceded by vkAllocateMemory() failed (error -1000072003),
              i.e. VK_ERROR_INVALID_EXTERNAL_HANDLE.
            vkcube --gpu_number 1 with
              LSFGVK_GPU='NVIDIA RTX A2000 Laptop GPU' — exit 0, clean.
          lsfg-vk defaults to the primary device; when the application is
          elsewhere, the shared external allocation fails and the
          application hangs.  Not a packaging defect, and Phase 5's
          selector VALUE for the laptop is now pre-validated.
          CONSEQUENCE: any test of frame generation in an application that
          is PRIME-offloaded to the dGPU MUST set the gpu key (conf.toml)
          or LSFGVK_GPU.  That includes Step 2.3's end-to-end Steam test on
          the laptop, which can set it by hand today; Phase 5 makes it
          declarative and machine-dispatched.  No phase reordering needed.

:: PHASE 2. STEAM SANDBOX INTEGRATION
================================================================================
    Goal: the layer is reachable inside nonguix' Steam container by
    container membership alone, and the superseded staging machinery is
    deleted.  No file is copied, no manifest is rewritten.
    - Step 2.1: Audit the container and sandbox surface (discovery only).
        Complete.  Results under DISCOVERY RESULTS.
    - Step 2.2: Add the container transformer to the package module.
        * A transformer taking a nonguix container and returning it with
          lsfg-vk among its 64-bit packages and the lsfg-vk variable
          regexps added to its preserved-env, through the public ngc-*
          accessors, with union32 pinned (I10).
          It transforms a CONTAINER, not Steam, so Heroic or any future
          container gains support for one line.
        * A Steam constructor parameterised by driver and by an additional
          passthrough set, re-entering composition at nonguix' public
          steam-container-for so upstream churn is absorbed at one site.
          Both parameters positional and required: define*-public is not
          bound in this context (I8), and a default driver would be
          silently wrong on one of the two machines.
        * No NVIDIA knowledge in this module: the driver and its
          passthrough set are supplied by the machine layer (D5).
        * nonguix must be a declared channel dependency (I9, D7).
        Test: the constructed Steam package evaluates and builds; its
        closure contains lsfg-vk exactly once; the 64-bit union's
        share/vulkan/implicit_layer.d contains the layer manifest, whose
        library_path names an output inside the container closure; the
        32-bit union contains NO lsfg-vk and no i686 qtdeclarative is
        attempted; the generated launcher's preserved-env contains
        lsfg-vk's regexps; the NVIDIA-driver variant evaluates with the
        passthrough set read per D5; and all of the above holds through an
        actual `guix pull', not only through -L (I9).
    - Step 2.3: Adopt the constructor per machine.
        * Desktop: the mesa-backed variant with an empty passthrough set.
        * Laptop: the NVIDIA-backed variant on the unversioned `nvda'
          alias — preserving the lockstep `steam-nvidia' had with
          nvidia-module-open — with nonguix' NVIDIA passthrough set read
          per D5 IN THE MACHINE FILE.
        * The channel must be committed, pushed and pulled before any
          reconfigure-based test (I6, I9).
        * The laptop's end-to-end test must set the gpu key for the dGPU,
          or it will reproduce OB2 inside the container.
        Test: both machines build; the deployed Steam's 64-bit union carries
        the manifest and its 32-bit union does not; a matching profile
        activates frame generation in a Steam game; DISABLE_LSFGVK in that
        game's launch options suppresses it.
    - Step 2.4: Remove the superseded staging machinery.
        * Delete the sandbox activation service from the home
          configuration in its entirety.
        * Remove the stale staged artifacts from the sandbox home.
        * Confirm no reference to the removed paths remains anywhere

        Test: Steam frame generation still works with the staged files
        deleted and the activation service gone.  Home builds.

:: PHASE 5. MACHINE DISPATCH IN THE HOME CONFIGURATION
================================================================================
    Goal: per-machine dispatch in the shared home configuration, and the GPU
    selector.  OB2 makes this load-bearing: without the selector, any
    application offloaded to the dGPU hangs.
    - Step 5.1: Introduce machine dispatch in the home configuration.
        * Mirror the system configuration's mechanism: machine identity with
          an environment-variable override so either machine's home
